## Setup Instructions

### Prerequisites

| Tool                   | Version       |
| ---------------------- | ------------- |
| Flutter                | ≥ 3.7.0       |
| Dart                   | ≥ 3.7.0       |
| Android Studio / Xcode | latest stable |

### 1. Clone & install dependencies

```bash
git clone <repo-url>
cd hypermart
flutter pub get
```

### 2. Firebase setup

The project uses Firebase Auth (phone OTP + Google Sign-In).

1. Create a Firebase project at [console.firebase.google.com](https://console.firebase.google.com).
2. Enable **Phone** and **Google** sign-in providers in _Authentication → Sign-in method_.
3. Register your Android app (`com.example.hypermart` or your bundle ID).
4. Download `google-services.json` and place it at:

   ```
   android/app/google-services.json
   ```

5. For iOS, download `GoogleService-Info.plist` and add it to `ios/Runner/` via Xcode.
6. The `lib/firebase_options.dart` file is auto-generated by the FlutterFire CLI:

   ```bash
   dart pub global activate flutterfire_cli
   flutterfire configure
   ```

### 3. Run the app

```bash
# Android
flutter run -d android

# iOS (macOS only)
flutter run -d ios

# Chrome (limited — no Firebase Phone Auth)
flutter run -d chrome
```

### 4. Build release APK

```bash
flutter build apk --release
# Output: build/app/outputs/flutter-apk/app-release.apk
```

### 5. Android signing (release builds)

Create a `key.properties` file in `android/` (do **not** commit this):

```properties
storePassword=<your-keystore-password>
keyPassword=<your-key-password>
keyAlias=<your-key-alias>
storeFile=<path-to-your-keystore.jks>
```

    ├── products/
    ├── cart/
    ├── wishlist/
    ├── orders/
    ├── location/

Each feature follows a strict three-layer structure:

```
feature/
├── domain/
│   ├── entities/       # Pure Dart value objects — zero framework imports
│   ├── repositories/   # Abstract repository interfaces
│   └── usecases/       # Single-responsibility business rules
├── data/
│   ├── models/         # JSON-serialisable data models
│   ├── mappers/        # Model ↔ Entity converters
│   ├── datasources/    # Remote (HTTP/Firebase) and local (SharedPreferences)
│   └── repositories/   # Concrete implementations of domain interfaces
└── presentation/
    ├── screens/        # Flutter UI widgets
    ├── controllers/    # Riverpod StateNotifiers (side-effect logic)
    ├── providers/      # Provider declarations & wiring
    └── widgets/        # Reusable UI components
```

## Architecture Explanation

HyperMart follows **Clean Architecture** as described by Robert C. Martin, adapted for Flutter.

### Dependency Rule

```
Presentation  ──→  Domain  ←──  Data
```

- **Domain** has zero Flutter or third-party dependencies. It owns entities, repository interfaces, and use cases.
- **Data** imports only the domain interfaces it implements plus the concrete SDKs (Firebase, HTTP).
- **Presentation** depends only on domain entities and use cases. It never imports Firebase, SharedPreferences, or any data-layer class directly.

### Layers

#### Domain Layer (`domain/`)

| Artefact               | Responsibility                                                                                                   |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `Entity`               | Immutable plain-Dart data class. Example: `UserEntity`, `ProductEntity`, `CartItemEntity`.                       |
| `Repository interface` | Abstract contract defining what data operations are available.                                                   |
| `UseCase`              | Single `call()` method. Orchestrates one business rule using a repository. Implements `UseCase<Return, Params>`. |

#### Data Layer (`data/`)

| Artefact         | Responsibility                                                                                                                     |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| `Model`          | JSON-aware DTO. Has `fromJson` / `toJson`. Never passed to the presentation layer.                                                 |
| `Mapper`         | Stateless converter: `Model → Entity` and `Entity → Model`.                                                                        |
| `DataSource`     | Talks to one external system only (Vendure API, Firebase, SharedPreferences).                                                      |
| `RepositoryImpl` | Implements the domain interface. Coordinates datasource + mapper. Catches SDK exceptions and re-throws as domain `Failure` values. |

#### Presentation Layer (`presentation/`)

| Artefact     | Responsibility                                                                                                |
| ------------ | ------------------------------------------------------------------------------------------------------------- |
| `Screen`     | Stateless (or `ConsumerWidget`) UI. Reads providers, dispatches actions.                                      |
| `Controller` | `StateNotifier` subclass. Holds transient UI state and calls use cases (or repository via Riverpod provider). |
| `Provider`   | Creates and wires dependencies through Riverpod.                                                              |

### Feature Dependency Graph

```
bootstrap.dart
  └─ ProviderScope
       ├─ sharedPreferencesProvider  (override with real SharedPreferences instance)
       ├─ authRepositoryProvider     ──→ AuthRepositoryImpl ──→ FirebaseAuth
       ├─ productRepositoryProvider  ──→ ProductRepositoryImpl ──→ ProductRemoteDataSource (Vendure)
       ├─ cartRepositoryProvider     ──→ CartRepositoryImpl ──→ CartLocalDataSource (SharedPrefs)
       ├─ wishlistRepositoryProvider ──→ WishlistRepositoryImpl ──→ WishlistLocalDataSource
       ├─ orderRepositoryProvider    ──→ OrderRepositoryImpl (in-memory + extendable)
       └─ locationRepositoryProvider ──→ LocationRepositoryImpl ──→ LocationService + GeocodingService
```

## State Management Approach

The app uses **Flutter Riverpod 2.x** throughout.

### Provider types used

| Type                          | Usage                                                                   |
| ----------------------------- | ----------------------------------------------------------------------- |
| `Provider<T>`                 | Synchronous singleton (repositories, services).                         |
| `StateNotifierProvider<N, S>` | Mutable UI state managed by a `StateNotifier` (cart, wishlist, orders). |
| `FutureProvider<T>`           | Async one-shot data fetch (product list, collections).                  |
| `StreamProvider<T>`           | Continuous stream (Firebase auth state → `authStateProvider`).          |
| `Provider.family<T, A>`       | Parameterised derived state (product by id, order by id).               |

### Auth state flow

```
FirebaseAuth.authStateChanges()
  └─ AuthRepositoryImpl.authStateChanges   (emits UserEntity?)
       └─ authStateProvider                (StreamProvider<UserEntity?>)
            └─ consumed by screens / profile
```

### Cart & Wishlist persistence pattern

Controllers use an **optimistic-update** pattern:

1. UI state is updated synchronously (instant feedback).
2. The repository persists to SharedPreferences asynchronously in the background.
3. On controller initialisation (`_load()`), persisted data is rehydrated from SharedPreferences.

This keeps the UI snappy while still surviving app restarts.

### Navigation

GoRouter is used with shell routes to support persistent bottom navigation. Route guards check `authStateProvider` to redirect unauthenticated users to the sign-in screen.

## Assumptions Made

| Area                            | Assumption                                                                                                                                                                                                                                                |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Back-end**                    | Products are served by a [Vendure](https://www.vendure.io/) instance over plain HTTP/GraphQL. The base URL is hard-coded inside `ProductRemoteDataSource`. Replace it with your own Vendure deployment URL.                                               |
| **Currency**                    | Prices are stored as minor currency units (paisa / cents). The display layer divides by 100 and formats as NPR (Nepalese Rupee) by default.                                                                                                               |
| **Geocoding**                   | The [Nominatim](https://nominatim.openstreetmap.org/) public API is used for reverse geocoding and place search. It is rate-limited to 1 request/second for non-commercial use. For production, replace with a paid provider (Google Maps, Mapbox, etc.). |
| **Orders persistence**          | `OrderRepositoryImpl` is an in-memory store (survives hot-reload but not app restart). It is intentionally kept simple — replace with a remote API or local database (sqflite / Isar) for production persistence.                                         |
| **Cart & Wishlist persistence** | Stored locally in SharedPreferences as JSON strings. No server-side cart sync is implemented.                                                                                                                                                             |
| **Phone OTP**                   | Phone number authentication uses Firebase Phone Auth. SMS delivery depends on your Firebase plan and country. Test phone numbers can be configured in the Firebase console.                                                                               |
| **Google Sign-In**              | Web sign-in uses `signInWithPopup`; Android/iOS uses the native Google Sign-In SDK. The SHA-1 fingerprint must be registered in your Firebase project for Android.                                                                                        |
| **Device Preview**              | `device_preview` is enabled in debug mode for quick multi-device testing. It is disabled in production builds.                                                                                                                                            |
| **Location permissions**        | The app requests `locationWhenInUse`. If the user denies permission, the location feature gracefully falls back to a manual address entry.                                                                                                                |
| **Image caching**               | `cached_network_image` is used for all remote images. No offline image cache invalidation strategy is implemented.                                                                                                                                        |
